Event standards

Five rules — and why
each one matters.

Every decision here is grounded in how Amplitude actually works. These aren't arbitrary preferences — they're what make the tool usable.

Rule 1 — Event naming: snake_case + object_action
✗  VideoVisit  ·  [WindowsPracticeApp] View Opened
✓  video_call_connected  ·  patient_file_viewed
Why

snake_case is Amplitude's own recommended convention. object_action matters because events sort alphabetically in Amplitude's schema view — grouping by object keeps related events together and makes the schema readable at a glance without needing to know the full history of every event name.

Rule 2 — Property naming: camelCase
✗  tenant_id  ·  app_version  ·  user_type
✓  tenantId  ·  appVersion  ·  userType
Why

Our SDKs are JavaScript and TypeScript — camelCase is the native convention. Mixing formats is exactly how gp:role and gp:user.role ended up as two separate properties meaning the same thing. That's a migration to fix. Consistent casing closes that door permanently.

Rule 3 — One event per action (Pattern A)
✗  VideoVisit + action: "started" / "muted" / "ended"
✓  video_call_connected  ·  video_call_end_call_pressed
Why

Amplitude builds every chart — funnels, retention, session replay filters, alerts — on event names, not property values. A property like action: "ended" is invisible to the funnel engine. Properties describe context. Event names describe what happened. That's the line.

Rule 4 — Five required properties on every event
tenantIdSegment by practice
userTypeSeparate provider vs patient
platformIsolate iOS vs Android vs Web
appVersionTie regressions to releases
userIdEnable per-user journey analysis
Why

Each property unlocks a specific axis of analysis. Without tenantId you can't segment by practice. Without platform, a bug affecting only iOS disappears into aggregated data. Without appVersion, QA can't tie a regression to a release. Remove any one and you lose an entire dimension across every event.

Rule 5 — Every event must be in schema before it fires
Description — what it tracks & when it fires
Category — feature area (e.g. Video Visits, Scheduling)
UX review required before any event reaches Engineering
Why

Most Practice App and Booking Portal events have no description and were never reviewed — they just started firing. Someone wrote a track() call, it shipped, and now we have 1.4M events/month from events nobody documented. The review step catches a wrong name or missing property before it ships — not after millions have fired.
